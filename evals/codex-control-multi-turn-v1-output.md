第一版的核心风险是：用户到底在不在乎“AI 表达风格优化”，而不是账号体系能不能成立。账号系统会引入注册登录、鉴权、数据存储、隐私和找回流程，却不能直接验证核心价值。

> 用户是否愿意反复使用它，把普通 AI 输出改成自己想要的表达风格？

账号、同步、社区都回答不了这个问题。它们只会让实验变量变多。

### Turn 2

**推荐定位**

> **给第一次开源的独立开发者用的 GitHub 发布前体检工具。**  
> 自动发现“还没准备好发布”的地方，并告诉用户现在最该补哪三项。

不要把它定位成“发布清单管理器”。清单本身很容易被文档、Notion 模板或普通 Todo 替代，现有开源发布清单甚至已有 50 项之多；GitHub 自己也提供 Community Standards 检查。[GitHub 官方文档](https://docs.github.com/en/communities/setting-up-your-project-for-healthy-contributions/creating-a-default-community-health-file)、[LaunchTry 清单](https://launchtry.com/resources/launch-checklist/open-source)

你的差异应该是：

- **不是给用户更多任务，而是判断哪些任务与当前仓库有关**
- **不是展示 30 项清单，而是指出发布阻塞项和下一步**
- **不是面向成熟维护者，而是降低第一次发布的不确定感**
- **不是项目管理工具，而是一次性的“发布准备度检查”**

**核心使用场景**

用户准备公开或推广 GitHub 项目时，在仓库页面打开扩展：

1. 扩展检查 README、LICENSE、安装说明、演示素材、贡献方式、Release 等。
2. 输出“可以发布 / 建议先修复”的判断。
3. 只突出三个最高优先级问题。
4. 每项说明为什么重要、如何完成，并保存本地进度。

**一句话文案**

> 第一次开源，不知道还漏了什么？在 GitHub 仓库里做一次发布前体检。

英文可以是：

> **Know what’s missing before you launch your first open-source project.**

**定位边界**

首版不要承诺：

- 帮项目获得更多 Star
- 管理长期版本发布
- 自动生成所有项目文档
- 覆盖 GitLab、npm、Product Hunt 等全部渠道
- 用一个模糊分数评判项目“好不好”

首版只承诺一个可验证结果：

> **用户在 10 分钟内知道：这个仓库能不能发，以及发布前最该修复什么。**

接下来需要确定定位更偏向哪种心理诉求：

- **A. 避免出丑和遗漏（推荐）**：强调“发布前体检”
- **B. 让项目看起来更专业**：强调“开源项目包装”
- **C. 帮助获得第一批用户**：强调“开源发布教练”

### Turn 3

优先查这几处，“偶尔”出现通常很像异步竞态：

1. **提交后是否真正更新了表单 state**
   - 受控表单的 `value` 是否仍绑定旧 state。
   - `reset()`、`setFormData()` 是否被后续请求覆盖。

2. **异步请求是否乱序返回**
   - 连续提交或重新拉取数据时，较早的请求可能更晚返回，把新值覆盖掉。
   - 给请求加序号或用 `AbortController` 取消旧请求。

3. **闭包拿到了旧 state**
   ```tsx
   setFormData(prev => ({ ...prev, name: newName }));
   ```
   尤其检查 `useCallback`、`useEffect`、定时器和事件监听器的依赖数组。

4. **缓存没有失效**
   - React Query：提交成功后检查 `invalidateQueries` / `setQueryData`。
   - SWR：检查 `mutate`。
   - Redux/Zustand：确认页面读取的 store 确实被更新。

5. **服务端响应或重新查询返回旧数据**
   - 提交响应成功，不代表紧接着的 GET 已读到新值。
   - 在 Network 面板对比请求 payload、响应内容和返回顺序。

6. **组件没有按预期重建**
   - `defaultValue` 只在首次挂载时生效；数据会变化时应使用 `value`。
   - 检查列表或表单组件的 `key` 是否稳定、是否错误复用了组件实例。

最快的定位方法是在“用户输入、提交开始、提交响应、重新拉取、写入 state”这几个节点打印时间戳和数据。若旧值是在某次响应后突然回来，基本就能锁定覆盖源。

### Turn 4

1. 看 Network：提交 payload 和响应是否都是新值。  
2. 查请求顺序：是否旧请求晚返回并覆盖新值。  
3. 查表单 state：提交后是否正确 `setState/reset`。  
4. 查 `useEffect/useCallback`：是否存在旧闭包或错误依赖。  
5. 查缓存：React Query/SWR/全局 store 是否及时更新或失效。  
6. 查组件绑定：是否误用 `defaultValue`，或 `key` 导致实例复用。

## Scenario 6

### Turn 1

这很像请求竞态：保存完成后触发了新列表请求，但更早发出的旧请求最后返回，反而把新状态覆盖了。也可能是缓存失效键不一致，不过目前还不能确认。

当前工作区只有 `calm-agent` 和文本挖掘文件，没有对应应用源码。请给我**项目绝对路径**，或贴出“保存记录”和“加载列表”的代码，我就能继续定位并直接修复。

### Turn 2

按目前仅有的现象“保存后偶发显示旧值”，我会这样排序：

1. **列表请求发生竞态，旧请求后返回**
   - 证据最强：“偶发”且回退到“旧值”很符合异步响应乱序。
   - 典型过程：旧列表请求 A 发出 → 保存成功 → 新请求 B 发出并返回新值 → A 最后返回，覆盖 B。
   - 验证：查看 Network 时间线和请求参数，确认最后完成的列表请求是否携带旧数据。

2. **前端缓存失效或重新拉取时序不正确**
   - 保存后可能没有 `await` 缓存失效/刷新，或失效的 query key 与列表实际 key 不一致。
   - React Query、SWR、RTK Query 一类方案尤其值得检查。
   - 验证：记录保存成功、invalidate、refetch、列表状态更新的时间与缓存键。

3. **后端缓存或只读副本存在短暂旧读**
   - 写入主库成功，但紧接着查询命中了缓存或延迟同步的只读副本。
   - 若手动刷新或等待几秒后自然恢复，这一方向会明显增强。
   - 验证：对比保存响应中的版本号/更新时间与随后列表接口返回值。

4. **保存接口过早返回**
   - 接口先返回“成功”，实际事务、队列任务或文件落盘尚未完成；前端立即查询时仍读到旧值。
   - 验证：直接在保存完成后连续调用列表接口，绕开前端状态层观察。

5. **本地状态被旧闭包或旧快照覆盖**
   - 例如保存后更新了某一项，随后 `useEffect`、分页合并或父组件旧 props 又重建了列表。
   - 验证：给每次 `setState` 标记来源，观察最后一次写入是谁触发的。

6. **并发保存或其他客户端覆盖**
   - 用户双击保存、自动保存并行、另一个页面同时编辑，都可能让较旧的写入最后落库。
   - 当前没有并发编辑证据，所以优先级较低。

第一步最值得做的是抓一次异常时的 Network 时间线。如果**列表接口本身返回旧值**，查后端缓存/一致性；如果**接口返回新值但页面显示旧值**，问题基本就在前端竞态或状态覆盖。

### Turn 3

这组证据已经基本排除“保存失败、数据库延迟、事务未提交”。问题范围明显收敛到前端读取与状态更新链路。

新的优先级：

1. **旧列表请求晚返回，覆盖了新值**
   - 当前最强方向，“偶发”尤其符合竞态。
   - PATCH 返回新值后，先前发出的 GET 仍可能携带旧快照，最后执行 `setState` 或写入查询缓存。
   - 重点看列表 GET 的**发起时间、完成时间和响应体**，不能只看 PATCH。

2. **PATCH 新值没有同步到列表使用的缓存**
   - 保存表单可能更新了详情状态，但列表读取的是另一份 query cache/store。
   - 随后的刷新没有执行、没有被 `await`，或 invalidate 使用了错误的 query key。
   - 若不重新进入页面就一直旧，这个方向会增强。

3. **页面存在多个状态来源，旧副本再次覆盖**
   - 例如列表缓存、组件本地 `useState`、父组件 props、分页数据各存一份。
   - PATCH 后其中一份更新，随后某个 `useEffect` 又用旧 props/cache 重置列表。

4. **更新列表时匹配不到对应记录**
   - 常见于 `id` 类型不同、使用了错误主键、分页/筛选参数不同，或者只更新了当前详情对象。
   - 如果只在某些记录、某些筛选条件下发生，应提高此项优先级。

5. **不可变更新或 memo 依赖错误**
   - 原地修改数组/对象、`useMemo` 缺依赖、`React.memo` 比较函数错误，可能导致数据已变但组件不重渲染。
   - 可用 React DevTools 判断组件拿到的 props/state 究竟是新值还是旧值。

下一步最有区分度：抓异常时**最后一个列表 GET 的响应体**。

- GET 返回旧值：查请求竞态、浏览器/API 缓存。
- GET 返回新值但 UI 旧：查本地状态覆盖、memo 和列表更新逻辑。
- PATCH 后没有 GET：查缓存失效以及是否把 PATCH 响应合并进列表缓存。

### Turn 4

现在最值得优先验证的是：

**标签页切换触发了重新取数或组件重挂载，随后旧缓存/旧请求结果覆盖了 PATCH 返回的新值。**

这仍是假设，但它最能同时解释：

- 只有“保存后立刻切换”更容易触发；
- PATCH 和数据库都已是新值；
- 页面显示旧值；
- 手动刷新后恢复。

优先检查标签切换时是否发生以下任一行为：

- 触发新的列表 GET；
- 保存前发出的 GET 在切换后才返回；
- 组件重新挂载，并从旧 query cache/store 初始化；
- `visibilitychange`、window focus 或 tab activation 触发自动 revalidate；
- 卸载阶段或 `useEffect` cleanup 后又写入旧状态。

最直接的验证方法是给每次列表请求加递增 `requestId`，记录：

```text
requestId / 发起时间 / 返回时间 / 返回记录值 / 写入状态时间
```

复现时如果看到较早请求最后写入，或组件重挂载后读取旧缓存，这个假设就得到支持。暂时不要先加延时或强制刷新，那会掩盖竞态。

### Turn 3

写入数据时，数据库不仅要更新表，还要同步维护相关索引。

主要开销：

- `INSERT`：向每个索引插入新条目。
- `UPDATE`：索引字段变化时，删除旧条目并插入新条目。
- `DELETE`：从每个索引移除对应条目。
- B+ 树索引可能触发页分裂、重平衡和额外 I/O。
- 唯一索引需要额外检查约束。
- 索引越多、索引键越大，写入开销越高。

本质：索引用额外的写入成本和存储空间换取更快的读取。
